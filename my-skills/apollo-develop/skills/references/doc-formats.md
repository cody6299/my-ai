# 本地文档格式规范

所有本地文档存储于 `${WORK_PATH_DOCUMENT}/AP-1234/`。

---

## analyse.md — 需求分析

记录需求分析内容，通过此文档必须能清楚表达本次要做什么以及怎么做。

**必须包含以下章节：**

- **基本信息**
  - Jira：`https://apollox.atlassian.net/browse/AP-1234`
  - 标题：从 Jira 获取的原始标题
  - 概述：一句话说明

- **需求背景**：详细描述背景、要做什么、要达到什么效果（≤6 句）

- **涉及服务**：本次需要修改哪些源码仓库，每个服务一句话概括功能

- **改动详情**：具体实现逻辑，从头到尾相互关联，逐步描述，清晰说明分哪几步完成、每步做什么（重点关注内容，必须完整）

---

## design.md — 需求设计

记录具体实现方案，含真实代码或伪代码，通过此文档必须能表达本次改动的具体内容。

**必须包含以下章节：**

- **涉及服务**：列出本次修改的服务（注意：服务与源码仓库不一定一一对应，一个仓库可能含多个服务）

- **涉及数据库**：新增表、修改表结构或数据；无则写"无"

- **涉及 Apollo 配置**：新增/修改/删除的配置项；无则写"无"

- **改动清单**：按服务列出修改了哪些具体文件及修改内容

- **调用链路**：完整调用链路（含改动前后对比）

- **注意事项**：需要特别注意的点；无则写"无"

---

## tasks.md — 子任务列表

记录本次开发拆分的子任务，需求开发流程按此逐步推进。

**格式：**

```markdown
## Tasks

- [ ] Task 1：新增 Entity（服务名/.../entity/XxxEntity.java）
- [ ] Task 2：新增 Mapper（服务名/.../mapper/XxxMapper.java）
- [ ] Task 3：实现 Service 逻辑
```

- 每行一个任务，包含单选框（`[ ]` 未完成，`[x]` 已完成）和简短描述
- 任务划分原则：相同模块放一起，从底层开始，每个 Task 完成后可独立运行验证

---

## summary.md — 开发总结（联调文档）

供联调和测试使用，说明如何部署、接口如何对接。

**必须包含以下章节：**

- **涉及服务**：格式为 `[服务名称](最新commitId链接): [PR](PR链接)`

- **改动说明**：≤5 句概述，然后按服务列出每个服务的改动说明

- **改动接口**：只列出 mgs 服务（对外网关）中的新增接口，供联调使用
  - 列出完整数据内容，不得省略
  - 数组至少展示两条数据

- **改动 DB**：对数据库的相关修改；无则写"无"

- **配置修改**：Apollo 配置的相关修改；无则写"无"

- **改动 XXL-JOB**：需要修改的定时任务；不存在则写"无"

---

## fix.{timestamp}.md — 问题修复记录

`{timestamp}` 为生成时的秒级时间戳（如 `fix.17952354.md`），可能存在多个。

**必须包含以下章节：**

- **问题描述**：清楚描述遇到了什么问题

- **涉及服务**：列出本次修改了哪些服务（链接指向本次修改的 commitId）

- **处理结果**：按服务列出每个服务中每个文件的修改内容

---

## publish.md — 发布文档

面向生产发布的文档，内容和逻辑要求严格，我将根据此文档操作发布服务。

**必须包含以下章节：**

- **涉及服务**：列出本次涉及的服务

- **改动说明**：简短说明本次改动

- **改动 DB**：需要修改的数据库（含 SQL 语句）；无则写"无"

- **配置修改**：需要修改的 Apollo 配置（含 Key、值、说明）；无则写"无"

- **改动 XXL-JOB**：需要修改的定时任务（含说明、执行频率、执行参数）；无则写"无"

- **发版顺序**：按顺序列出发布步骤，例如：
  1. 执行 DB 变更 SQL
  2. 修改 Apollo 配置
  3. 发布 cloud-future-depositwithdraw 服务
  4. 更新依赖服务的 Jar 包
  5. 发布 cloud-mgs-depositwithdraw 服务
  6. 配置 XXL-JOB 定时任务

---

## process.md — 进度状态

```text
analyse:done
design:done
develop:in_progress
test:done
finish:done
```

流程名称：`analyse` / `design` / `develop` / `test` / `finish`
状态：`in_progress`（进行中）/ `done`（已完成）

---

## memory.md — 上下文记忆

内容和格式由 AI 自行决定，无需用户确认即可修改。

**保证内容包含：**
- 当前进度和状态
- 关键背景知识（涉及服务、重要结论）
- 与用户讨论达成的关键结论
- 重要注意事项

每次进入会话时加载此文件，快速恢复上下文，继续工作。
