# 各流程详细说明

## 需求开始

确定 JiraId 后，在 `${WORK_PATH_DOCUMENT}` 下创建以 JiraId 命名的子目录（如 `AP-1234`），并在其中创建空文件 `process.md`。

- 若目录和 `process.md` 已存在：不输出任何内容，直接进入「需求分析」
- 若本次新建：输出 `开始开发 AP-1234 的功能`，然后直接进入「需求分析」（无需等待用户确认）

---

## 需求分析

**目标**：理解需求，与用户讨论确认实现方向，输出 `analyse.md`。

**步骤：**

1. **获取信息**：通过 JiraId 从 Jira 读取需求文档，重点关注标题和评论（尤其是用户的评论）。用户评论不一定正确，需综合分析后与用户讨论
2. **深入理解**：查阅 `${WORK_PATH_KNOWAGE}` 知识库，在源码仓库中检索关键字，深入理解背景
3. **沟通确认**：不确定的地方向用户询问，经多次交互最终确认需求内容和实现方向

确认后，向用户输出简短说明（≤5 句），**必须取得用户同意**，再生成 `analyse.md`。

生成后：
```
需求分析文档已生成，文档路径：./AP-1234/analyse.md，请查看
```
执行 `open ./AP-1234/analyse.md` 自动打开文档。

用户要求调整时，每次完成后输出：`需求分析文档已更新，请查看`

用户最终确认无误后，更新 `process.md`（`analyse:done`），询问：
```
需求分析已完成，是否继续「需求设计」流程？
```

---

## 需求设计

**目标**：给出具体实现方案（含伪代码或真实代码），拆分子任务，输出 `design.md` 和 `tasks.md`。

**步骤：**

1. 依赖 `analyse.md`，对每个详细设计与用户交互确认
2. 将需求拆分成多个子任务（每个任务功能完整、改动适中，从底层开始，每个 Task 可独立运行验证）

拆分完成后，向用户输出简短说明（列出子任务及简单描述，每个任务 ≤3 句）。**必须取得用户同意**，再生成 `design.md` 和 `tasks.md`。

生成后：
```
需求设计文档已生成，文档路径：./AP-1234/design.md
本次需求被拆分成 3 个子任务，任务文档已生成，文档路径：./AP-1234/tasks.md
```
分别用 `open` 命令打开两个文档。

用户确认无误后，更新 `process.md`（`design:done`），询问：
```
需求设计已完成，是否继续「需求开发」流程？
```

---

## 需求开发

### 分支检查（每次进入此流程必须执行）

确认每个相关源码仓库是否在正确分支（以 JiraId 为分支名）：

- **无对应分支**：先切换到 master → 拉取最新 → 创建新分支
  ```
  已为 cloud-future-depositwithdraw 服务创建新分支：AP-1234
  ```
- **分支已存在**：切换到对应分支
  ```
  已将 cloud-future-depositwithdraw 服务切换到分支：AP-1234
  ```
- **已在正确分支**：
  ```
  cloud-future-depositwithdraw 服务已在分支：AP-1234
  ```

### 任务执行

按 `tasks.md` 顺序逐个执行。每个任务开始前输出：
```
任务(1/5)：初始化实体类及 Mapper
是否开始？
```

完成后输出：
```
任务(1/5)：初始化实体类及 Mapper 已完成
请 Review 后告知结果
```

根据用户反馈：
- **代码有问题需调整**：确认内容后修改，输出：`修改完成：调整了实体类的字段，请 Review`
- **用户已自行调整**：查看修改点，确认无问题后输出：`修改确认：调整了实体类字段，无影响。提交代码还是继续？`

用户确认无误后，更新 `tasks.md` 将任务设为已完成，然后输出：
```
任务(1/5) 初始化实体类及 Mapper 已全部完成，下一个任务是：
任务(2/5) 定义 Service 相关服务，提交代码还是继续？
```

- **用户希望提交**：本地提交所有未提交的代码，输出：`代码已提交，commit: 03513521，继续开发下一个任务`
- **用户不提交**：直接进入下一个任务

### 收尾工作（所有任务完成后）

输出所有涉及服务和 PR 所需信息（Title + Description），要求用户手动提交 PR：
```
需求 AP-1234 已完成，涉及服务：
- cloud-future-depositwithdraw
- cloud-mgs-depositwithdraw

请手动提交 PR 后，将 PR 链接告诉我。

**提交 PR 所需信息：**

**cloud-future-deposit-withdraw：**
Title: `AP-1234: Change Deposit Risk Rule`
Description:
...
```

收到 PR 链接后，描述 `summary.md` 内容（≤5 句），询问：`是否生成 summary.md 文件？`

用户确认后生成 `summary.md`，用 `open` 打开，等待用户 Review。调整完成后输出：
```
需求开发流程已完成，各文档路径：
- analyse.md：./AP-1234/analyse.md
- design.md：./AP-1234/design.md
- tasks.md：./AP-1234/tasks.md
- summary.md：./AP-1234/summary.md

是否上传到 Confluence？
```

### 上传至 Confluence（用户确认后）

在远程根路径下创建子目录，名称格式：`AP-1234: Change Entity`（JiraId + 简短描述，≤20 字符）

| 本地文件 | 远程文档名 |
|----------|------------|
| `analyse.md` | AP-1234:开发总结(AI) |
| `design.md` + `tasks.md` | AP-1234:开发设计(AI) |
| `summary.md` | AP-1234:联调文档(AI) |

上传完成后输出远程文档链接，等待用户确认。确认无误后，更新 `process.md`（`develop:done`），输出：
```
需求开发流程已完成，请联调测试，有问题继续与我沟通。
```

---

## 需求测试

**目标**：解决联调/测试中发现的问题。

**步骤：**

1. 确认相关源码仓库都在正确分支，有调整必须输出说明
2. 输出：`需求开发已完成，是否在联调测试中遇到了什么问题？`
3. 通过交互确认问题，修改代码后输出：`代码已修改，请 Review`
4. 用户确认后询问：`问题已修复，是否提交代码？`
5. 用户确认提交后：
   - 以当前秒级时间戳生成 `fix.{timestamp}.md`（包含问题描述、解决方案、commitId）
   - 输出：`解决方案已提交，本地文档：./AP-1234/fix.17952354.md，是否还有其他问题？`

---

## 需求完成

**目标**：整理发布文档，标记需求完成。

**步骤：**

1. 确认各源码仓库处于正确分支且分支干净（无未提交改动）
2. 输出：`需求完成，准备生成部署文档`
3. 生成 `publish.md`（包含 DB 变更、配置修改、定时任务、发版顺序），输出：
   ```
   发布文档已生成：./AP-1234/publish.md
   是否上传到 Confluence？
   ```
4. 用户确认后上传，输出 Confluence 链接，等待用户确认
5. 确认无误后，更新 `process.md`（追加 `finish:done`），输出：
   ```
   需求 AP-1234 开发结束，请根据发布文档发布服务。
   ```
